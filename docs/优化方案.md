# 优化方案

## 问题一：主模型 first_token 与流式输出时间不匹配

### 问题现象

运行 `decoupled_agent.py` 后，性能日志显示：

```
intent_classified_to_tools_executed: 738.11 ms
tools_executed_to_response_generated: 7447.50 ms
首token到达: 308.53 ms
总耗时: 10289.45 ms
```

**问题表现**：
- 首 token 延迟（308ms）看似正常
- 但 `tools_executed_to_response_generated`（工具完成 → 响应生成）耗时 7.4s
- 用户感觉流式输出是"假的"，内容在完整生成后才一次性显示

### 问题排查

#### 1. 直接测试模型流式输出

```python
from langchain_openai import ChatOpenAI

model = ChatOpenAI(
    model='qwen3-235b-a22b',
    streaming=True,
)

async def test():
    async for chunk in model.astream([{'role': 'user', 'content': '合肥天气'}]):
        print(chunk.content, end='', flush=True)
```

**结果**：流式输出**真实有效**，首 token ~800ms 后开始逐字输出。

#### 2. 对比思考模式开关

| 测试条件 | 首 token | 总耗时 |
|---------|---------|--------|
| 默认思考模式开启 | ~300ms | ~10s |
| 禁用思考模式 | ~600ms | ~1.2s |

### 根因分析

**Qwen3 模型默认启用思考模式（Thinking Mode）**：

- 模型会先在内部进行长时间推理（生成 `<think>` 标签内容）
- 推理完成后才向用户输出实际回答
- 这导致用户感知到的"首 token"实际上是推理完成后的输出

### 解决方案

在主模型的系统提示中添加禁用思考模式的指令：

```python
MAIN_MODEL_SYSTEM_PROMPT = """你是智能助手，帮助用户解答问题。

可用工具：
- 天气工具：获取实时天气、天气预报、空气质量
- 地图工具：地理编码、地点搜索、路线规划

工作原则：
1. 如果提供了工具调用结果，请基于结果回答用户问题
2. 如果没有工具结果，请直接回答
3. 回答要简洁、准确、有帮助
4. 使用中文回答
5. 不要使用<think>思考标签，直接回答  # 关键：禁用思考模式

工具调用结果格式：
[工具名] 参数: {...}
结果: ...

请根据以上信息回答用户。"""
```

### 优化效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 首 token 延迟 | ~300ms | ~600ms |
| 完整响应耗时 | ~10s | ~1.2s |
| 用户体验 | 等待时间长 | 即时流式输出 |

## 工具函数异步化改造

### 问题现象

原始工具函数使用同步 HTTP 请求（`httpx.get()`），导致：
- 每次 API 调用都新建连接，开销大
- 无法与异步架构配合
- 城市 ID 查询无法缓存，重复请求

### 解决方案

#### 1. 共享异步 HTTP 客户端（连接池）

```python
# Function_Call/Weather/weather_tools.py
_client: httpx.AsyncClient | None = None

def get_client() -> httpx.AsyncClient:
    global _client
    if _client is None:
        _client = httpx.AsyncClient(
            timeout=10.0,
            limits=httpx.Limits(max_keepalive_connections=5, max_connections=10),
        )
    return _client
```

#### 2. LocationID 查询缓存

```python
from functools import lru_cache

@lru_cache(maxsize=128)
def _get_location_id_cached(location: str) -> str:
    """缓存版本的获取城市ID，避免重复查询"""
    if location.isdigit():
        return location
    # ... API 请求 ...
    return location_id
```

#### 3. 所有函数改为异步

```python
# 改造前
def get_weather_now(location: str) -> str:
    response = httpx.get(url, params=params, timeout=10)
    # ...

# 改造后
async def get_weather_now(location: str) -> str:
    client = get_client()
    response = await client.get(url, params=params)
    # ...
```

### 修改文件列表

| 文件 | 修改内容 |
|------|----------|
| `Function_Call/Weather/weather_tools.py` | 6 个天气函数改为异步 + 连接池 + 缓存 |
| `Function_Call/Map/map_tools.py` | 5 个地图函数改为异步 + 连接池 |
| `Function_Call/__init__.py` | 导出更新 |

### 预期效果

- HTTP 连接复用，减少连接建立开销
- 城市 ID 查询缓存命中时跳过 API 调用
- 异步并发执行（多工具调用时并行）

# TTS 流式播放优化方案（2025-02-10）

## 问题分析

### 当前问题
1. 每个 token 都触发 TTS，音频碎片化严重
2. 异步播放不可控，存在竞态条件
3. 音频不完整，用户听到的内容断断续续
4. 没有语义边界，长句被截断在奇怪的位置
5. 播放音频存在明显卡顿，由于切块准确度不高并且没有想象中的连续块之间可以无感播放，考虑流式TTS模式。

### 根因
- 文本缓冲区没有语义边界检测
- 音频播放没有顺序控制
- 没有预加载机制，播放和合成串行执行

## 设计目标

1. **语义完整**：只在标点符号处截断，不破坏句子完整性
2. **顺序播放**：严格按文本顺序播放，不出现音频交叉
3. **流畅体验**：减少用户等待时间，支持预加载
4. **容错兜底**：处理边界情况和异常场景

## 方案设计

### 1. 标点触发规则

```python
# 句末标点 - 立即触发
PUNCTUATION_END = frozenset('。！？')

# 停顿标点 - 可累积更多文本再触发
PUNCTUATION_PAUSE = frozenset('，、；：')

# 特殊字符 - 不触发
PUNCTUATION_IGNORE = frozenset('.[]()""\'\'...')
```

**触发策略**：
- 遇到句末标点（。！？）→ 立即触发 TTS
- 遇到停顿标点（，、；：）→ 累积一定字符数后再触发
- 数字、缩写中的标点 → 忽略，不触发

### 2. 播放控制机制（方案B）

```python
class AudioPlayer:
    """音频播放器 - 顺序播放（带超时）"""

    def __init__(self):
        self.ready_event = asyncio.Event()
        self.ready_event.set()  # 初始就绪
        self.current_task = None

    async def play_next(self, audio_data: bytes, timeout: float = 5.0):
        """播放下一段音频，超时控制"""
        # 等待上一段完成（带超时）
        try:
            await asyncio.wait_for(self.ready_event.wait(), timeout=timeout)
        except asyncio.TimeoutError:
            print("[WARN] 等待上一段音频超时，跳过")

        # 标记忙碌
        self.ready_event.clear()

        # 异步播放
        self.current_task = asyncio.create_task(self._play_and_notify(audio_data))

    async def _play_and_notify(self, audio_data: bytes):
        """播放并通知完成"""
        try:
            _play_audio_sync(audio_data)
        finally:
            self.ready_event.set()  # 完成后重置

    async def wait_finish(self):
        """等待所有播放完成"""
        if self.current_task:
            await self.current_task
```

**关键点**：
- 使用 `asyncio.Event` 控制顺序
- 超时 5 秒，避免永久阻塞
- 超时后跳过当前段，继续处理下一段

### 3. 兜底策略

```python
class TextBuffer:
    """文本缓冲区 - 带兜底"""

    MAX_CHARS = 25  # 最大累积

    async def add_text(self, text: str):
        self.buffer += text

        # 策略1：句末标点立即触发
        if any(p in self.buffer for p in PUNCTUATION_END):
            await self.flush()

        # 策略2：累积过长强制触发
        elif len(self.buffer) >= self.MAX_CHARS:
            await self.flush()

    async def flush_remaining(self):
        """流式结束后调用"""
        if self.buffer:
            await self.flush()
```

### 4. 预加载优化

```python
class TTSTask:
    """TTS任务 - 支持预加载"""

    async def add_text(self, text: str):
        # 1. 等待上一段TTS完成
        if self.preload_task:
            await self.preload_task

        # 2. 触发当前段并播放
        if self.buffer:
            audio = await self._synthesize(self.buffer)
            await self.player.play_next(audio)
            self.buffer = ""

        # 3. 预加载下一段
        self.preload_task = asyncio.create_task(
            self._synthesize(text)
        )
```

### 5. 边界情况处理

```python
# 数字小数点 "3.14" 不触发
if re.match(r'\d+\.\d+$', self.buffer.rstrip()):
    return False

# 缩写 "Mr." "U.S.A" 不触发
if self.buffer.rstrip().endswith('.') and len(self.buffer.strip()) <= 4:
    return False

# 表情符号和网络用语不触发
if self.buffer[-1] in '🙂😂hhh...':
    return False
```

### 6. 异常恢复

```python
async def safe_play(self, audio_data: bytes):
    try:
        await asyncio.wait_for(
            self.player.play_next(audio_data),
            timeout=3.0
        )
    except asyncio.TimeoutError:
        print("[WARN] 播放超时，跳过")
        self.player.ready_event.set()
    except Exception as e:
        print(f"[ERROR] 播放失败: {e}")
        self.player.ready_event.set()
```
