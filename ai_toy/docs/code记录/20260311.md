# 2026年3月11日 用户画像智能体重构记录

## 一、问题描述

### 1.1 用户画像更新识别不准确

- **现象**：用户画像守护进程（ProfileDaemon）严重依赖对特定关键词的识别
- **问题**：如果没有匹配到关键词，就认为不需要记录，局限性太高
- **用户表达**：
  - "我最喜欢画画了" → ❌ 不匹配"喜欢"开头
  - "我最近在学钢琴" → ❌ 不是"喜欢"
  - "我周末都去踢足球" → ❌ 行为，非声明
  - "妈妈说我很内向" → ❌ 第三方提及
  - "我喜欢收集邮票" → ❌ 关键词列表中没有

### 1.2 问题根因

原有实现采用**关键词列表 + 正则匹配**的方式：
- `hobby_keywords` 硬编码了运动、阅读、绘画等分类
- 依赖 `r"^我喜欢(.+)"` 等固定正则表达式
- 关键词不在列表中就无法识别

## 二、方案对比

### 方案1：关键词列表（原有方案）

```python
hobby_keywords = {
    "运动": ["跑步", "打球", "游泳", "爬山", "骑车", "足球", "篮球", "运动"],
    "阅读": ["看书", "读书", "阅读", "故事", "绘本", "漫画"],
    # ...
}
```

**优点**：
- 快速、确定性
- 无需额外 LLM 调用

**缺点**：
- 需要维护关键词列表
- 无法识别列表外的新偏好
- 无法理解语义相似的表达

---

### 方案2：Embedding 向量相似度（备选方案）

**思路**：
1. 将用户输入转换为向量
2. 与预定义的偏好类别向量进行相似度比较
3. 超过阈值则识别为偏好表达

**优点**：
- 无需维护关键词列表
- 支持语义相似的表达
- 可处理未见过的偏好类型

**缺点**：
- 需要 embedding 模型支持
- 向量存储和计算有资源开销
- 阈值需要调优

---

### 方案3：LLM 判断（最终方案）

**思路**：
1. 用户画像智能体（ProfileAgent）判断当前输入是否存在偏好表达
2. LLM 自动提取偏好信息，无需关键词列表
3. LLM 自动去重，判断是否已在当前画像中

**优点**：
- 灵活、通用
- 无需维护关键词列表
- 支持任意类型的偏好表达
- 自动语义去重

**缺点**：
- 每次更新需要额外一次 LLM 调用

**选择理由**：儿童陪伴产品对话风格自由灵活，LLM 判断最合适，且减少硬件依赖负荷。

## 三、修改内容

### 3.1 新建文件

#### `Function_Call/profile_agents.py`

创建用户画像智能体，参考天气/地图 Agent 模式：

```python
class ProfileAgent:
    """用户画像更新智能体"""

    async def run(self, user_input: str, requery_params: Dict) -> Dict[str, Any]:
        """执行智能体判断"""
        # 1. 获取当前用户画像
        current_profile = get_current_profile()

        # 2. 构建 prompt，包含当前画像用于去重
        prompt_content = self.system_prompt.replace(
            "{current_profile}",
            json.dumps(current_profile, ensure_ascii=False, indent=2)
        )

        # 3. LLM 判断是否需要更新
        response = await self.model.ainvoke(messages)
        result = json.loads(response.content)

        # 4. 如果需要更新，执行更新逻辑
        if result.get("use_tool", False):
            result["result"] = await self._do_update_profile(...)

        return result
```

**核心逻辑**：
- `run()` 方法：LLM 判断是否需要更新
- `_do_update_profile()` 方法：LLM 提取偏好信息并更新画像

## 四、新逻辑流程

```
用户输入 → ReQuery阶段 
         → ToolAgentExecutor.execute_all() 
         → ProfileAgent.run() 
            → LLM判断是否需要更新 + 自动去重
         → _do_update_profile()
            → LLM提取偏好信息
            → 更新用户画像
         → 保存到文件
```

**关键点**：
1. **Agent 判断层**：判断是否需要执行更新
2. **去重判断**：LLM 自动检查当前画像是否已存在
3. **提取层**：LLM 提取具体偏好信息（姓名、爱好、性格等）
4. **更新层**：追加新值到画像，去重

## 六、方案对比总结

| 方案 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| 关键词列表 | 硬编码正则匹配 | 快速确定 | 需维护、无法识别新偏好 |
| Embedding | 向量相似度 | 语义理解 | 需要 embedding 模型 |
| **LLM判断** | 两次LLM调用 | 灵活通用、自动去重 | 额外调用开销 |

**选择 LLM 方案的原因**：
1. 儿童陪伴产品对话风格自由
2. 无需维护关键词列表
3. 自动语义去重
4. 无需额外硬件（embedding 模型）

## 七、后续优化方向

### 7.1 性能优化

- 考虑缓存判断结果，减少重复 LLM 调用
- 可以设置"冷却时间"，同一用户短时间内不重复触发

### 7.2 字段扩展

- 当前支持：name, hobbies, likes, character, age, gender, location
- 可扩展：dislikes（不喜欢的事物）、relationship 等

### 7.3 冲突处理

- 当前使用追加去重策略
- 可考虑：时间戳优先级、置信度评分等更复杂策略

### 8.2 JSON 解析兼容

LLM 可能返回两种格式：
```python
# 格式1：嵌套
{"updates": {"hobbies": "跑步"}}

# 格式2：直接
{"hobbies": "跑步"}
```

需要兼容两种格式。
