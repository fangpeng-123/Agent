# TTS语音播放问题排查与修复记录

## 问题描述

用户报告：模型回答后TTS服务播放的语音顺序错乱，例如提问"讲一个成语故事吧"，模型回复完整内容但语音播放时先播放最后一句，然后跳回前面播放。

## 问题排查

### 1. 初步分析

通过探索代码库，发现TTS相关文件：
- `src/core/agent.py` - 核心TTS播放逻辑（TTSTask类）
- `src/services/tts_service.py` - TTS服务封装
- `src/services/tts/base.py` - TTSProviderBase基类

### 2. 发现的根本原因

#### 问题1：异步任务调度问题
```python
# agent.py 第676行（原始代码）
asyncio.create_task(tts_task.add_text(text))
```
使用`asyncio.create_task()`是非阻塞的，导致多个文本块可能不按顺序进入TTS队列。

#### 问题2：过早触发合成
每次LLM输出一个小片段就立即放入队列并清空缓冲区，导致多个小片段独立并行合成，合成完成的顺序不确定。

#### 问题3：音频播放被中断
```python
# 原始代码
sd.play(data, framerate)
# 不等待播放完成，立即返回
```
`sd.play()`后没有调用`sd.wait()`，导致新音频会覆盖正在播放的旧音频，造成"突然中断"。

## 尝试的解决方案

### 方案1：首句快速响应 + 后续句累积（已实施）

**思路**：
- 首句：8字就触发合成（快速响应用户）
- 后续句：累积50字再触发（减少切分次数）

**实施的修改**：

1. **TTSTask类添加首句标志**
```python
self._is_first_segment = True  # 首句标志
```

2. **触发阈值**
```python
FLUSH_MIN_LENGTH_FIRST = 8   # 首句触发阈值
FLUSH_MIN_LENGTH_REST = 50   # 后续句触发阈值
FLUSH_PUNCTUATIONS = ("。", "？", "！", "；")  # 句末标点
```

3. **add_text方法改进**
```python
async def add_text(self, text: str):
    # 累积到缓冲区
    self._text_buffer += text
    
    # 根据是否是首句选择不同的阈值
    is_first = self._is_first_segment
    min_length = FLUSH_MIN_LENGTH_FIRST if is_first else FLUSH_MIN_LENGTH_REST
    
    # 遇句末标点或累积足够长度时触发
    if self._text_buffer[-1] in FLUSH_PUNCTUATIONS or len(self._text_buffer) >= min_length:
        await self._text_queue.put(self._text_buffer)
        if is_first:
            self._is_first_segment = False
```

4. **分段逻辑**
```python
def _split_text_segments(self, text: str) -> list[str]:
    # 遇句末标点立即分割
    # 遇句中标点且>=25字可分割
    # >=100字强制分割，避免在词语中间切断
```

5. **累积播放**
```python
# AudioPlayer累积100KB音频后播放
MAX_BUFFER_SIZE = 100 * 1024
```

### 方案2：真正的流式播放（未实施）

**思路**：使用`on_audio`回调，收到音频块立即播放，不等待整段合成完。

**原因**：后来用户希望保留当前版本的详细日志，故未实施。

## 最终修复

### 关键修复：添加sd.wait()

```python
# 修改前
sd.play(data, framerate)
# 不等待播放完成，立即返回

# 修改后
sd.play(data, framerate)
sd.wait()  # 等待播放完成，避免新音频中断当前音频
```

这解决了每段音频末尾突然中断的问题。
