## 一、问题列表

### 1. 消息列表中缺少上下文信息
- **现象**：运行 `decoupled_agent.py` 时，Message List 中本应包含的 context info 字段消失了
- **预期内容**：天气、地点、日期等预加载信息应该传入模型
- **原因**：`ContextManager` 从未被初始化

### 2. 延迟统计不准确
- **现象**：用户期望的是"从问题输入开始到语音播报开始"的时间，但终端只打印了 TTFA 和 TTS 耗时
- **原因**：缺少端到端延迟指标

## 二、实现的修改

### 修改1：修复上下文信息丢失问题

**问题根因**：
- `builder.py` 第 112-114 行调用 `get_context_info()` 获取上下文信息
- 但 `decoupled_agent.py` 只调用了 `agent.init_user_profile()`，这**不会**初始化 context manager
- `context_manager.py` 中的 `init_context()` 方法从未被调用
- 因此 `get_context_info()` 返回空字符串

**修复方案**：在 `init_user_profile()` 中添加对 `init_context()` 的调用

**文件**：`src/core/agent.py`

```python
async def init_user_profile(self):
    """初始化用户画像和上下文（在程序启动时调用）"""
    try:
        from Function_Call.UserProfile.profile_daemon import get_daemon
        from Function_Call.UserProfile.user_profile_tools import USER_PROFILES

        daemon = get_daemon()
        await daemon.start()
        print(...)
    except Exception as e:
        print(f"[WARN] 初始化用户画像失败: {e}")

    # 初始化上下文管理器（地点、天气、日期时间）
    await self.init_context()

async def init_context(self):
    """初始化上下文管理器"""
    try:
        from src.utils.context_manager import init_user_context
        await init_user_context("user_001")
        print("[OK] 上下文管理器已初始化 (地点、天气、日期时间)")
    except Exception as e:
        print(f"[WARN] 初始化上下文管理器失败: {e}")
```

### 修改2：添加端到端延迟统计

**问题根因**：
- 原有的 `TTFA(首段音频延迟)` 是从 **LLM开始生成** 到 **首段音频播放**
- 缺少从 **用户输入结束** 到 **首段音频播放** 的端到端指标

**修复方案**：
1. 在 `PerformanceMetrics` 中添加 `end_to_end_latency` 属性
2. 在 `decoupled_agent.py` 中记录用户输入结束时间
3. 在 `agent.py` 中计算端到端延迟

## 四、TTS延迟优化讨论

### 4.1 当前架构问题

**问题**：段与段之间的延迟主要来自 TTS 服务

当前使用的 `TTSTask` 架构：
```
LLM输出文本 → 累积到8字 → 发送请求 → 等待音频 → 播放
                                      ↓
                              每段都需要重新建立 WebSocket 连接！
```

**每段的延迟 = 网络延迟 + WebSocket握手 + TTS合成 + 返回音频**

### 4.2 已有的优化方案（但未使用）

项目已经实现了一套**更好的流式方案**：

| 当前使用 | 已实现但未使用 |
|---------|--------------|
| `QwenTTSService` (段式) | `StreamTTSService` (真流式) |
| 每段重建连接 | WebSocket 连接复用 |
| 累积8字触发 | 真正的流式输出 |

### 4.3 优化方案

**方案 1：使用现有的 StreamTTSService（推荐，最小改动）**
- 项目已经实现了真正的流式 TTS
- 支持 WebSocket 连接复用
- 边听边合成下一段

**方案 2：优化 TTSTask 的分段策略**
- 首段更短（4字 → 更快开始）
- 预启动下一段的 TTS 请求

**方案 3：改用已有的 StreamTTSService**
- 修改 agent.py 使用 `StreamTTSService.process_llm_stream()`
- 这是真正的流式架构
