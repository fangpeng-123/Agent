# 2026年3月10日 队列管理和播放顺序问题修复记录

## 一、问题描述

### 1. 跨段播放 + 跳回原顺序
- **现象**：播放音频时出现跨段播放，然后又跳回原来的顺序播放
- **用户反馈**：听起来顺序不对

### 2. 音频被截断
- **现象**：单个段太长时会被截断，直接播放下一段音频
- **原因**：TTS合成在响应生成完成后才开始，而不是流式输出期间并行执行

### 3. 日志混乱
- **现象**：控制台日志输出格式不统一，难以追踪播放顺序

---

## 二、问题根因分析

### 问题1：`_finished` 标志设置时机过早

**错误代码位置**：`finish()` 方法

```python
# 错误：先设置 _finished，再刷新文本
async def finish(self):
    self._finished = True  # ❌ 过早设置，导致 AudioTask 提前退出
    
    # 刷新剩余文本时，_finished 已经为 True
    async with self._text_buffer_lock:
        if self._text_buffer:
            await self._text_queue.put(self._text_buffer)  # 可能被忽略
```

**后果**：
- `_audio_player_task` 检测到 `_finished=True` 且队列为空时立即退出
- 但 TTS 消费者可能还在处理文本，导致部分音频丢失

### 问题2：任务退出逻辑有竞态条件

**错误代码位置**：`_audio_player_task` 方法

```python
# 原代码：超时退出逻辑不完善
while True:
    try:
        audio_data = await asyncio.wait_for(
            self._audio_queue.get(), timeout=0.1
        )
    except asyncio.TimeoutError:
        # 问题：仅仅因为超时就检查退出，可能误判
        if self._finished and self._audio_queue.empty():
            break  # ❌ 可能还有音频在合成中！
        continue
```

**后果**：
- 超时 0.1 秒后就检查是否应该退出
- 但 TTS 合成可能需要更长时间，导致音频丢失

### 问题3：日志格式混乱

**问题**：
- TTS 日志和 AudioPlayer 日志混在一起
- 难以追踪播放顺序
- 音频大小用 bytes 表示，不直观

## 三、修复方案

### 修复1：延迟设置 `_finished` 标志

**修改位置**：`finish()` 方法

```python
# 修复后：先刷新文本，再设置 _finished
async def finish(self):
    # 1. 刷新剩余文本（不设置 _finished）
    async with self._text_buffer_lock:
        if self._text_buffer:
            await self._text_queue.put(self._text_buffer)

    # 2. 发送结束信号
    await self._text_queue.put(None)

    # 3. 现在才设置 _finished - 确保所有文本都已入队
    self._finished = True

    # 4. 等待 TTS 合成完成
    if self._tts_task:
        await self._tts_task
```

### 修复2：改进任务退出逻辑

**修改位置**：`_audio_player_task` 方法

```python
# 修复后：只有收到 None 结束信号才退出
while True:
    try:
        audio_data = await asyncio.wait_for(
            self._audio_queue.get(), timeout=0.2
        )
    except asyncio.TimeoutError:
        continue  # 超时不代表退出，继续等待

    if audio_data is None:
        # 收到结束信号后，等待确认没有新的合成结果
        await asyncio.sleep(0.3)
        if self._audio_queue.empty():
            break  # 确认队列为空才退出
```

### 修复3：整理日志输出格式

**修改位置**：`_tts_consumer`、`_audio_player_task`、`_play_loop`

```python
# TTS 合成日志
print(f"\n--- [TTS Chunk #1] 合成中 (长度=23) ---")
print(f"    文本: 警察的工作是保护大家的安全...")
print(f"[TTS Chunk #1] >> 合成成功 199KB")

# 音频队列日志
print(f"\n=== [AudioTask] 启动 ===")
print(f"[AudioTask] >> 收到音频 199KB")

# 播放日志
print(f"=== [AudioPlayer] 播放循环启动 ===")
print(f"[AudioPlayer #1] >> 播放 199KB")
print(f"[AudioPlayer #1] >> 播放完成")
```

## 七、2026年3月10日 下午 补充

### 验证确认

经过详细日志分析，**播放顺序实际上是正确的**：

1. TTS 消费者按文本累积策略依次触发合成（#1 → #2 → #3 → #4）
2. 音频按合成完成的顺序放入队列
3. AudioPlayer 按队列顺序依次播放

用户听到的"跨段播放"可能是因为：
- 队列积压：合成速度快于播放速度，多段音频在队列中等待
- 听觉错觉：长音频片段之间的间隔不够明显

### 日志整理

将原有的混乱日志整理为统一格式：
- `[TTS Chunk #N]` - TTS 合成进度
- `[AudioTask]` - 音频队列任务
- `[AudioPlayer #N]` - 实际播放进度

音频大小统一使用 KB 表示，更易阅读。

## 八、技术总结

### 双队列架构

```
LLM流式输出 → _text_queue → TTS合成 → _audio_queue → AudioPlayer → 播放
```

### 关键设计要点

1. **文本累积策略**：按标点和长度智能分句
2. **任务并行**：TTS 合成与播放并行执行
3. **退出信号**：使用 None 作为队列结束标志
4. **状态同步**：通过 `_finished` 标志协调任务退出

### 注意事项

1. `_finished` 必须在所有文本入队后再设置
2. 退出前需要等待确认队列已清空
3. 日志对调试至关重要，需要保持清晰可读

## 九、2026年3月10日 晚间 文本切块策略优化

### 9.1 问题描述

#### 问题1：句子被错误截断
- **现象**：完整句子被从中间截断，例如：
  - 输入："小伙伴！你觉得排队时最有趣的事情是什么呀？"
  - 输出（错误）："小伙伴！你觉得排" + "推不挤..."
- **原因**：`MAX_CHUNK_LENGTH = 30` 设置过小，句子在中间被强制触发

#### 问题2：两层切分导致语义混乱
- **现象**：agent.py 切分一次，base.py 又二次切分
- **原因**：
  1. agent.py 按 30 字触发
  2. StreamTTS/base.py 又按标点二次切分
  3. 导致完整句子被拆散

#### 问题3：日志不清晰
- **现象**：无法追踪具体切了几块、每块内容
- **原因**：日志只显示前50字符，截断过多

### 9.6 待优化问题

#### 问题1：首段太长导致回复速度慢
- **现象**：如果第一段累积到 60 字，需要等待整段 TTS 合成完才能播放
- **影响**：用户等待时间长
- **解决方案**：
  - 方案A：分段并行合成（第一段累积到15字就触发）
  - 方案B：快速首段（前N字先合成让用户先听到）
  - 方案C：保持现状（简单但首段慢）

#### 问题2：并行合成可能导致顺序错乱
- **现象**：如果开启真正的并行合成，不同长度块完成时间不同
- **原因**：队列特性导致先完成的后播放
- **解决方案**：需要序号标记 + 按序号排序播放

#### 问题3：base.py 死代码
- **现象**：代码中存在 236-301 行的重复/废弃代码
- **影响**：代码可读性差
- **解决方案**：清理死代码

#### 问题4：用户画像的守护进程记录不准确
- **现象**：用户画像相关的守护进程（daemon）日志记录不完整或不准确
- **影响**：难以追踪用户画像的状态变化和问题排查
- **可能原因**：
  1. 守护进程日志级别设置不当
  2. 状态变更未完整记录
  3. 异常信息捕获不全面
- **解决方案**：
  - 方案A：完善守护进程的状态转换日志
  - 方案B：增加关键操作的生命周期日志
  - 方案C：添加结构化日志便于分析

### 9.7 技术总结

#### 两层切分流程（优化后）
```
LLM 流式输出
    ↓
agent.py 缓冲区（按句子/60字触发）
    ↓ 输出完整句子
StreamTTS/base.py
    ↓ 整句保护：≤30字+句末标点 → 直接返回
    ↓ 否则：按标点切块/长句拆分
TTS 合成
```

#### 关键设计要点

1. **优先句子边界**：遇句末标点立即触发，保证语义完整
2. **长文本保护**：来自上游的长文本不再二次切分
3. **阈值调整**：MAX_CHUNK_LENGTH 30→60，避免句子截断
4. **日志优化**：显示完整文本，便于调试
