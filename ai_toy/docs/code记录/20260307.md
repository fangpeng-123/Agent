## 问题一：Qwen3 模型 enable_thinking 错误

### 问题描述
运行项目时报错：
```
Error code: 400 - {'error': {'message': 'parameter.enable_thinking must be set to false for non-streaming calls', 'type': 'invalid_request_error', 'param': ...
```

### 问题成因分析

**技术背景**：
- Qwen3 模型引入了 `enable_thinking` 参数，用于启用/禁用思考模式
- 思考模式是 Qwen3 的特性，允许模型在回答前进行更长时间的推理
- 该参数在流式调用（streaming）时是可选项，但在非流式调用（non-streaming）时**必须显式设置为 false**

**问题根因**：
项目中使用了以下 Qwen3 模型进行非流式调用（`ainvoke`）：
- `qwen3-30b-a3b`（天气、地图、日期时间智能体）
- `qwen3-30b-a3b-instruct-2507`（ReQuery 意图识别）
- `qwen3-30b-a3b`（用户画像 AI 更新）

这些模型在通过 LangChain 的 `ChatOpenAI` 调用时，默认会传递 `enable_thinking=true`，导致与阿里云 DashScope API 的非流式调用产生冲突。

**影响范围**：
- ReQuery 意图识别失败
- 所有工具智能体（天气、地图、日期时间）无法执行
- 用户画像更新失败

### 修复内容
在以下 5 个文件的 `ChatOpenAI` 初始化中添加 `extra_body={"enable_thinking": False}`：

| 文件 | 行号 |
|------|------|
| `src/core/requery.py` | 34 |
| `Function_Call/weather_agents.py` | 23 |
| `Function_Call/datetime_agents.py` | 23 |
| `Function_Call/map_agents.py` | 23 |
| `Function_Call/UserProfile/profile_ai_tools.py` | 164 |

### 修复代码示例
```python
# 修复前
return ChatOpenAI(
    model="qwen3-30b-a3b",
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    api_key=SecretStr(api_key) if api_key else None,
    temperature=0.1,
)

# 修复后
return ChatOpenAI(
    model="qwen3-30b-a3b",
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    api_key=SecretStr(api_key) if api_key else None,
    temperature=0.1,
    extra_body={"enable_thinking": False},
)
```

## 问题二：一次对话后需要停顿很久才能输入下一条消息

### 问题描述
用户反馈：一次对话后需要停顿很久才可以输出下一个问题。

### 问题成因分析

**架构背景**：
系统采用三阶段流水线架构：
1. ReQuery 意图识别
2. 工具智能体并行执行
3. 主模型流式回复 + TTS 语音合成

**TTS 播放流程**：
```
LLM 流式输出
    │
    ▼
add_text() → 文本缓冲区（遇到标点触发合成）
    │
    ▼
_tts_consumer() → TTS 合成
    │
    ▼
_audio_player_task() → 放入播放队列
    │
    ▼
AudioPlayer._play_loop() → 播放音频
    │
    ▼
sd.play() + sd.wait() → 同步阻塞播放
```

**问题根因**：
在 `src/core/agent.py` 中，`_play_audio_sync` 函数使用了**同步阻塞模式**：

```python
def _play_audio_sync(audio_data: bytes):
    # ...
    sd.play(data, framerate)
    sd.wait()      # ← 阻塞等待音频播放完成！
    sd.stop()      # ← 停止播放器
```

当 `TTSTask.finish()` 被调用时：
```python
async def finish(self):
    # ...
    await self._tts_task          # 等待 TTS 合成
    await self._play_task         # 等待播放任务
    await self.audio_player.wait_finish(timeout=30.0)  # ← 最多等待 30 秒！
    await self.audio_player.stop()
```

**影响**：
- 用户必须等待所有音频播放完毕（可能 30 秒以上）才能输入下一条消息
- 严重影响交互体验

### 修复内容

#### 修改 1：替换播放函数（第 41-75 行）
将 `_play_audio_sync` 改为 `_play_audio_async`，移除 `sd.wait()` 和 `sd.stop()`：

```python
def _play_audio_async(audio_data: bytes):
    """异步播放音频（不阻塞，立即返回）"""
    # ... 音频处理逻辑 ...
    sd.play(data, framerate)
    # 不等待播放完成，立即返回
```

#### 修改 2：修改 `_play_loop` 方法（第 106 行）
```python
# 将
_play_audio_sync(audio_data)
# 改为
_play_audio_async(audio_data)
```

#### 修改 3：修改 `TTSTask.finish()` 方法
优化结束逻辑，确保队列清空后再停止：

```python
async def finish(self):
    """结束，等待所有音频处理完成"""
    self._finished = True
    
    # 1. 刷新剩余文本
    # 2. 发送结束信号
    # 3. 等待 TTS 合成完成
    # 4. 等待音频队列清空（关键！最多 1 秒）
    for _ in range(10):
        if self._audio_queue.empty():
            break
        await asyncio.sleep(0.1)
    
    # 5. 等待播放队列清空（最多 10 秒）
    await self.audio_player.wait_finish(timeout=10.0)
    
    # 6. 最后才停止
    await self.audio_player.stop()
```

#### 修改 4：主流程调用（第 509-511 行）
添加注释说明异步播放机制。

## 问题三：吞音问题

### 问题描述
修改异步播放后，出现音频尾部被吞掉的问题。

### 问题成因分析

**问题根因**：
修改异步播放后，`finish()` 方法的停止逻辑存在问题：

```python
async def finish(self):
    self._finished = True
    # 刷新剩余文本...
    await self._text_queue.put(None)
    
    # 等待 TTS 合成
    if self._tts_task:
        await self._tts_task
    
    # ❌ 立即停止播放器！此时可能还有音频未处理
    await self.audio_player.stop()
```

**具体问题**：
1. `_audio_queue` 中可能还有未处理的音频
2. `_audio_player_task` 可能还没把音频放入 `audio_player.queue`
3. `stop()` 会立即中断正在播放的音频

**架构流程图**：
```
LLM 输出
   │
   ▼
┌──────────────────┐
│  add_text()      │ ← 文本累积，遇到标点触发
│  (文本缓冲区)    │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  _tts_consumer   │ ← 合成音频
│  (TTS 合成)      │
└────────┬─────────┘
         │ 音频片段
         ▼
┌──────────────────┐
│ _audio_player    │ ← 取出音频，放入播放队列
│    _task         │        ▲
└────────┬─────────┘         │
         │                  │
         ▼                  │
┌──────────────────┐        │
│  AudioPlayer     │────────┘
│  _play_loop      │ ← 播放音频
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  sd.play()      │ ← 吞音发生点！
└──────────────────┘

问题：finish() 调用 stop() 时，_audio_queue 中可能还有未处理的音频
```

**影响**：
- 音频尾部被截断
- 用户听到的语音不完整

### 修复内容
优化 `finish()` 方法的停止逻辑：
- 增加等待音频队列清空的循环（最多 1 秒）
- 等待播放队列清空后再停止（最多 10 秒）

```python
async def finish(self):
    """结束，等待所有音频处理完成"""
    self._finished = True
    
    # 1. 刷新剩余文本
    # 2. 发送结束信号
    # 3. 等待 TTS 合成完成
    # 4. 等待音频队列清空（关键！最多 1 秒）
    for _ in range(10):
        if self._audio_queue.empty():
            break
        await asyncio.sleep(0.1)
    
    # 5. 等待播放队列清空（最多 10 秒）
    await self.audio_player.wait_finish(timeout=10.0)
    
    # 6. 最后才停止
    await self.audio_player.stop()
```
