**创建日期**: 2026年2月6日  
**测试环境**: Windows 10, Python 3.10, Conda agent_test

## 一、项目概述

本项目实现了一个基于 LangChain 1.x 的智能体，集成了百度地图 MCP 服务和和风天气 MCP 服务，可以通过自然语言对话查询地理位置和天气信息。

### 涉及文件

- `agent_test.py` - 主程序，智能体入口
- `MCP/Map/server.py` - 百度地图 MCP 服务（5个工具）
- `MCP/Weather/server.py` - 和风天气 MCP 服务（6个工具）
- `.env` - 环境变量配置文件

### 问题2: Agent 导入失败

**现象**:
```python

from langchain.agents import create_tool_calling_agent, AgentExecutor

```

**原因**: LangChain 1.x 移除了旧的 Agent API，统一使用 `create_agent`

**解决**: 更新代码使用新的 API
```python
from langchain.agents import create_agent
agent = create_agent(model=model, tools=tools, system_prompt="...")
```

### 问题3: MCP 工具调用失败（核心问题）

**现象**:
```
[ERROR] 处理请求时出错: anyio.ClosedResourceError
```

**详细堆栈**:
```
File "langchain_mcp_adapters/tools.py", line 395, in execute_tool
    call_tool_result = await session.call_tool(...)
anyio.ClosedResourceError
```

**根本原因分析**:

最初的代码结构：
```python
# 错误示例
async def load_tools():
    async with client.session("map") as session:
        tools = await load_mcp_tools(session)  # 加载工具
    return tools  # 会话已关闭！

# 后续调用工具时
await tool.ainvoke({"location": "北京"})  # ❌ 会话已关闭，报错
```

**问题**: `async with` 块结束后，MCP 会话自动关闭。工具对象内部引用了已关闭的会话，导致调用时抛出 `ClosedResourceError`。

---

## 三、解决方案

### 方案1: 使用 client.get_tools() 方法（推荐）

**优点**:
- 自动管理会话生命周期
- 代码简洁
- 官方推荐方式（langchain-mcp-adapters 0.2.1+）

**实现代码**:
```python
from langchain_mcp_adapters.client import MultiServerMCPClient
from langchain.agents import create_agent

# 配置 MCP 服务器
server_config = {
    "map": {
        "transport": "stdio",
        "command": sys.executable,
        "args": [MCP_MAP_SERVER_PATH],
    },
    "weather": {
        "transport": "stdio",
        "command": sys.executable,
        "args": [MCP_WEATHER_SERVER_PATH],
    },
}

# 创建客户端
client = MultiServerMCPClient(server_config)

# 关键：使用 client.get_tools() 加载工具
# 这会保持会话在整个 Agent 生命周期内打开
all_tools = await client.get_tools()

# 创建 Agent
agent = create_agent(
    model=model,
    tools=all_tools,
    system_prompt="You are a helpful assistant...",
)

# 调用 Agent（工具调用时会话仍然打开）
response = await agent.ainvoke({"messages": messages})
```

### 方案2: 使用会话上下文管理器

**适用场景**: 需要手动控制会话生命周期

**实现代码**:
```python
@asynccontextmanager
async def mcp_session():
    client = MultiServerMCPClient(server_config)
    try:
        async with client.session("map") as map_session:
            async with client.session("weather") as weather_session:
                # 加载工具
                map_tools = await load_mcp_tools(map_session)
                weather_tools = await load_mcp_tools(weather_session)
                # 创建 Agent
                agent = create_agent(..., tools=map_tools + weather_tools)
                # yield 控制权（会话保持打开）
                yield agent
    finally:
        # 清理资源
        pass

# 使用
async with mcp_session() as agent:
    await run_conversation(agent)  # 整个对话期间会话保持打开
```

## 八、重要注意事项

1. **会话生命周期**: MCP 工具必须在会话打开时调用，使用 `client.get_tools()` 可自动管理

2. **环境变量**: MCP 服务作为独立子进程启动，需要确保它们能正确加载 `.env` 文件

3. **API 密钥**: 确保三个密钥（API_KEY、BAIDU_MAP_AK、HEFENG_KEY）都正确配置

4. **并发限制**: `client.get_tools()` 会启动所有服务器，确保系统资源充足

5. **编码问题**: Windows 终端可能有编码问题，建议使用 UTF-8 编码

## 十一、Weather MCP 服务测试问题

**测试日期**: 2026年2月6日  
**问题名称**: Weather MCP 服务测试失败（FunctionTool 不可调用、API Host 迁移、Windows 编码问题）

### 问题描述

在运行 `MCP/Weather/test_weather.py` 测试脚本时，遇到以下三个主要问题：

**问题1: FunctionTool 对象不可调用**
```
错误: 'FunctionTool' object is not callable
```

**问题2: API 调用失败（403 Invalid Host）**
```
Status: 403
Response: {"error":{"status":403,"type":"...#invalid-host","title":"Invalid Host"}}
```

**问题3: Windows 控制台编码错误**
```
UnicodeEncodeError: 'gbk' codec can't encode character '\U0001f4cd'
```

### 根本原因分析

#### 问题1: FastMCP 2.x 版本兼容性问题
FastMCP 2.x 的 `@mcp.tool()` 装饰器将函数包装成了 `FunctionTool` 对象，不再是可直接调用的函数。测试代码直接调用被装饰的函数导致错误。

**错误代码示例**:
```python
@mcp.tool()
def get_weather_now(location: str) -> str:
    # 业务逻辑...
    pass

# 测试代码中直接调用
result = get_weather_now("北京")  # ❌ FunctionTool 对象不可调用
```

#### 问题2: 和风天气 API Host 变更
和风天气从 2026 年起逐步停止原公共 API 地址（`geoapi.qweather.com`, `devapi.qweather.com`）的服务，改为每个开发者使用独立的 API Host（如 `abc1234xyz.def.qweatherapi.com`）。继续使用旧地址会导致 403 Invalid Host 错误。

#### 问题3: Windows 终端 GBK 编码限制
代码中使用了大量 emoji 字符（如 📍🌡️🌤️ 等），Windows 终端默认使用 GBK 编码，无法显示这些 Unicode 字符，导致 `UnicodeEncodeError`。

### 解决方案

#### 方案1: 业务逻辑与 MCP 装饰器分离
将业务逻辑提取为内部函数，装饰后的函数仅作为包装器：

```python
def _get_weather_now(location: str) -> str:
    """业务逻辑实现"""
    # 实际的 API 调用和数据处理逻辑
    pass

@mcp.tool()
def get_weather_now(location: str) -> str:
    """MCP 工具入口"""
    return _get_weather_now(location)
```

**测试代码更新**:
```python
# 导入内部函数进行测试
from server import (
    _get_weather_now as get_weather_now,
    _get_weather_forecast as get_weather_forecast,
    # ...
)
```

#### 方案2: 支持新的 API Host 配置
修改 `server.py` 添加 `HEFENG_API_HOST` 环境变量支持：

```python
HEFENG_KEY = os.getenv("HEFENG_KEY")
HEFENG_API_HOST = os.getenv("HEFENG_API_HOST")

# 使用新的API Host（如果配置了）或回退到旧地址
if HEFENG_API_HOST:
    BASE_URL = f"https://{HEFENG_API_HOST}/v7"
    GEO_URL = f"https://{HEFENG_API_HOST}/geo/v2"
else:
    # 旧版公共API地址（2026年起逐步停止服务）
    BASE_URL = "https://devapi.qweather.com/v7"
    GEO_URL = "https://geoapi.qweather.com/geo/v2"
```

**.env 配置更新**:
```env
HEFENG_KEY="your_api_key"
HEFENG_API_HOST="ne4nmu6n5v.re.qweatherapi.com"
```

**获取 API Host**:
1. 登录和风天气控制台 (https://console.qweather.com/)
2. 进入"设置"页面
3. 复制你的 API Host（格式如 `abc1234xyz.def.qweatherapi.com`）

#### 方案3: 移除 Emoji 字符
将所有 emoji 替换为纯文本标记：

```python
# 修改前
return (
    f"📍 {location} 实时天气\n"
    f"🌡️ 温度：{now.get('temp', '未知')}°C\n"
    f"🌤️ 天气：{now.get('text', '未知')}\n"
    # ...
)

# 修改后
return (
    f"[位置] {location} 实时天气\n"
    f"[温度] {now.get('temp', '未知')}°C\n"
    f"[天气] {now.get('text', '未知')}\n"
    # ...
)
```

### 经验教训

1. **FastMCP 版本兼容性**: FastMCP 2.x 的装饰器模式改变了函数的可调用性，需要将业务逻辑与 MCP 注册分离

2. **第三方 API 变更**: 和风天气 API Host 的变更是 2026 年的重大变更，需要及时更新配置

3. **跨平台兼容性**: Windows 终端默认 GBK 编码对 Unicode 字符支持有限，开发时需要考虑平台差异

4. **环境变量管理**: MCP 服务作为独立子进程运行，需要确保 `.env` 文件路径正确，建议使用 `Path(__file__).parent` 动态定位
