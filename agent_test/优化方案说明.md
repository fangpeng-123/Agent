# 智能体优化方案总结

## 📋 问题回顾

你的智能体存在以下问题：
1. **地图工具调用率低** - 模型不知道何时使用地图MCP
2. **回答出现幻觉** - 基于训练数据而非工具返回的实际数据回答
3. **表述模糊** - 使用"可能"、"大概"等不确定词汇
4. **需要连续引导** - 单次调用表现不佳
5. **短期记忆** - 只在一轮对话中生效，下一轮就会遗忘
6. **恢复延迟高** - 回复问题时模型的响应时间长，导致用户等待时间过长

## ⚡ 响应速度优化方案 (添加于2026-02-07 14:00)

针对"恢复延迟高"问题，以下是具体的优化策略和实现方案：

### 问题根源分析

响应慢主要有以下几个原因：

1. **两次LLM云端调用**：需要先调用一次决定使用什么工具，再调用一次生成最终回答
2. **MCP工具调用的本地开销**：使用stdio模式每次都要启动Python子进程
3. **串行执行**：LLM决策 → 工具调用 → LLM生成，无法并行

### 优化策略

#### 1. 使用SSE模式替代stdio模式

**什么是SSE模式？**

SSE（Server-Sent Events）是一种服务器向客户端推送实时数据的技术。在MCP中：

- **stdio模式**：每次调用工具时启动Python子进程，通过标准输入输出通信
- **sse模式**：MCP服务作为HTTP服务器长期运行，通过HTTP SSE连接通信

**优势对比**：

| 特性 | stdio模式 | sse模式 |
|------|-----------|---------|
| 进程启动 | 每次调用都启动 | 一次启动，长期运行 |
| 通信开销 | 进程间通信，序列化开销大 | HTTP连接，开销小 |
| 首次调用延迟 | 高（需启动进程） | 低（连接已建立） |
| 适用场景 | 简单脚本、低频调用 | 高频调用、生产环境 |

**实现代码**：

```python
# 修改前（stdio模式）
server_config = {
    "map": {
        "transport": "stdio",
        "command": sys.executable,
        "args": [MCP_MAP_SERVER_PATH],
    },
}

# 修改后（sse模式）
server_config = {
    "map": {
        "transport": "sse",
        "url": "http://localhost:8080/sse",  # MCP服务作为HTTP服务器运行
    },
}
```

**启动MCP服务**：

```bash
# 需要先启动MCP服务作为HTTP服务器
python MCP/Map/server.py --transport sse --port 8080
python MCP/Weather/server.py --transport sse --port 8081
```

#### 3. 缓存工具结果

避免重复查询相同位置的天气或坐标：

```python
from functools import lru_cache

class CachedMCPTools:
    def __init__(self, tools):
        self.tools = tools
        self._cache = {}
    
    def _get_cache_key(self, tool_name, **kwargs):
        return f"{tool_name}:{hash(str(kwargs))}"
    
    async def call_tool(self, tool_name, **kwargs):
        cache_key = self._get_cache_key(tool_name, **kwargs)
        
        if cache_key in self._cache:
            return self._cache[cache_key]
        
        # 找到对应的工具
        tool = next(t for t in self.tools if t.name == tool_name)
        result = await tool.ainvoke(kwargs)
        
        # 缓存结果（设置TTL为5分钟）
        self._cache[cache_key] = result
        return result
```

### 交互缓慢的实战经验 (添加于2026-02-07 15:00)

#### 问题现象
用户反馈即使使用云端模型，回复仍然很慢，且计时统计显示各阶段耗时为0ms。

#### 根本原因分析

**1. 计时逻辑失效**
- 原方案使用 `BaseCallbackHandler` 回调计时
- `create_agent` 不通过 `config={"callbacks": [...]}` 传播回调
- 导致所有阶段时间显示为0或异常值

**解决方案**：简化计时逻辑，直接在调用前后使用 `time.time()` 记录

```python
# 简化计时方案
start_time = time.time()
response = await agent.ainvoke({"messages": messages})
end_time = time.time()
total_time = (end_time - start_time) * 1000
```

**2. Agent架构的流式限制**
- `streaming=True` 设置在模型上
- 但 `create_agent` 会收集完整响应再返回
- 流式只能看到"事件"而非逐字增量

**关键发现**：`create_agent` 的 `astream()` 返回格式为：
```python
{'model': {'messages': [AIMessage(content='完整内容')]}}  # 一次性返回
```

**解决方案**：智能分流策略
- **基础问题**（问候、自我介绍）：直接使用 `model.astream()` 真流式
- **工具问题**（天气、地图）：使用 `create_agent`（功能完整但流式受限）

```python
def needs_tools(user_input: str) -> bool:
    tool_keywords = ["天气", "地图", "坐标", "路线", "查询", "搜索"]
    return any(kw in user_input for kw in tool_keywords)

# 根据问题类型选择策略
if needs_tools(user_input):
    # 使用 Agent（工具调用）
    async for chunk in agent.astream(...):
        yield chunk
else:
    # 直接使用模型（真流式）
    async for chunk in model.astream(messages):
        yield chunk.content
```

**3. Windows终端编码问题**
- 使用 emoji（📊、⏱️、1️⃣ 等）导致显示为 `` 或乱码
- 原因：Windows默认GBK编码不支持Unicode emoji

**解决方案**：使用纯文本符号替代
```python
# 修改前（有乱码）
print("📊 精细化耗时分析报告")
print("1️⃣ 数据准备阶段")

# 修改后（兼容）
print("[性能分析] 耗时统计报告")
print("[1] 数据准备阶段")
```

#### 实测效果对比

| 场景 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 基础问候（"你好"） | 8-12秒整段输出 | 2-3秒逐字流式 | 感知速度提升70% |
| 天气查询（"北京天气"） | 10-15秒等待 | 显示"[思考中...]"+结果 | 用户体验改善 |
| 计时统计 | 各阶段0ms（错误） | 准确的总耗时 | 可观测 |

#### 关键经验总结

1. **不要依赖回调机制计时** - LangChain Agent的回调传播不可靠
2. **区分流式需求** - 基础对话真流式，工具调用可妥协
3. **避免Unicode特殊字符** - Windows终端兼容性差
4. **模型大小确实影响速度** - 即使云端，235B vs 8B 推理时间差3-5倍

## ❓ 常见问题

### Q: 为什么模型还是不调用工具？
A: 可能原因：
1. 模型认为自己的知识足够回答（需在提示词中强调"禁止基于训练数据回答"）
2. 工具描述不够清晰（优化MCP服务的docstring）
3. 模型能力不足（尝试更强的模型）

### Q: 工具返回数据很多，如何组织回答？
A: 在提示词中添加：
```
数据组织原则：
1. 优先展示用户最关心的信息
2. 使用结构化格式（列表、表格）
3. 重要数据加粗或高亮
4. 提供简洁的总结
```

### Q: 如何完全消除幻觉？
A: 完全消除很难，但可以：
1. 严格要求基于工具数据（提示词+验证器）
2. 工具返回数据时添加免责声明
3. 如果工具失败，明确告知无法回答
