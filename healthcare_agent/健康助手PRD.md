# 健康管理服务 PRD

## 1. 项目概述

### 1.1 产品定位

基于企业微信的 AI 健康管理助手，为用户提供体检报告解读、健康咨询、个性化健康建议服务。

### 1.2 核心价值

- 降低健康咨询门槛，提供 7x24h AI 健康助手
- 体检报告结构化解读，生成用户健康画像
- 通过企微实现便捷的用户触达和交互

### 1.3 目标用户

- 对健康管理有需求的个人用户
- 付费会员制，<100 用户规模验证

### 1.4 关键决策汇总

本节汇总了项目讨论过程中确定的关键决策：

#### 1.4.1 企微接入方式

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 接入方式 | 企业微信应用（自建 API 对接） | 实现完整功能，支持用户识别、私聊、会员管理 |
| 替代方案 | 暂不考虑企微机器人 | 能力受限，无法实现核心需求 |
| 资质要求 | 需要企业微信认证 | 企业营业执照、域名备案、HTTPS 证书 |

#### 1.4.2 客服审核方案

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 审核界面 | 企微消息推送 | 无需开发额外后台，客服通过企微即可审核 |
| 交互方式 | 文本指令（通过/驳回） | 简单易用，无需学习成本 |
| 审核级别 | L0 直接回复，L1/L2/L3 推送客服 | 分级处理，平衡效率与安全 |

#### 1.4.3 通知服务方案

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 主要方式 | 客服手动提醒 | 灵活度高，成本低 |
| 辅助方式 | 自动消息推送 | 报告完成、异常指标等场景自动触发 |
| 技术实现 | APScheduler 定时任务 | 轻量易用，无需额外基础设施 |

#### 1.4.4 会员支付

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 当前方案 | 暂不考虑 | MVP 阶段聚焦核心功能 |
| 后续计划 | 人工处理或接入微信支付 | 验证商业模式后再接入 |

#### 2.1.4 企微 API 对接关键实现

```python
class WeChatWorkMessageHandler:
    """企业微信消息处理器"""

    async def handle_callback(self, request):
        """
        处理企微消息回调
        1. 验证消息签名
        2. 解密消息内容
        3. 解析消息类型
        4. 路由到对应处理器
        """
        # Step 1: 验证签名
        msg_signature = self._verify_signature(request)

        # Step 2: 解密消息
        encrypted_xml = await request.body()
        decrypted_content = self._decrypt_message(encrypted_xml)

        # Step 3: 解析消息
        msg_type = decrypted_content.get('MsgType')
        from_user = decrypted_content.get('FromUserName')

        # Step 4: 路由处理
        handlers = {
            'text': self._handle_text,
            'image': self._handle_image,
            'file': self._handle_file,
            'event': self._handle_event
        }
        return await handlers[msg_type](decrypted_content)

    async def send_message(self, user_id, message):
        """
        发送消息给用户
        使用企微消息推送 API
        """
        url = f"{self.api_base_url}/cgi-bin/message/send"
        data = {
            "touser": user_id,
            "msgtype": "text",
            "agentid": self.agent_id,
            "text": {"content": message},
            "safe": 0
        }
        return await self._post(url, data)
```

### 2.2 企微应用创建前提条件

创建企业微信应用需要以下资质：

| 资质要求 | 说明 | 优先级 |
|----------|------|--------|
| 企业微信认证 | 需要企业营业执照完成认证 | P0 |
| 企业主体账号 | 需要企业作为主体注册企微 | P0 |
| 域名备案 | 回调 URL 需要 ICP 备案 | P1 |
| HTTPS 证书 | 回调 URL 需要 HTTPS 访问 | P1 |

如果暂时没有企业资质，可考虑以下替代方案：

| 替代方案 | 优点 | 缺点 |
|----------|------|------|
| 微信服务号 | 门槛较低，个人可申请 | 功能受限，无法实现所有需求 |
| Telegram Bot | 无需企业资质 | 用户使用门槛高 |
| Web 应用 | 完全自主可控 | 用户触达困难 |

**建议：优先解决企业资质问题，长期来看企业微信应用是最佳选择。**

## 3. 功能模块详细设计

### 3.1 消息接入层 (企微应用 API 自建)

#### 3.1.1 功能描述

通过自建 API 直接对接企业微信应用，实现企微消息的收发和路由。

#### 3.1.2 企微应用 vs 机器人 选择说明

本项目选择企业微信应用，原因如下：

| 需求 | 企微机器人 | 企微应用 | 本项目选择 |
|------|------------|----------|------------|
| 接收用户消息 | 仅群聊@机器人 | 支持私聊+群聊 | ✅ 企微应用 |
| 识别用户身份 | 无法识别 | 可识别具体用户 | ✅ 企微应用 |
| 1v1 专属对话 | 不支持 | 支持 | ✅ 企微应用 |
| 会员权益管理 | 无法实现 | 可实现 | ✅ 企微应用 |
| 客服审核消息 | 无法推送 | 支持推送 | ✅ 企微应用 |
| 实现复杂度 | 简单 | 稍复杂 | 值得投入 |

#### 3.1.3 交互模式

| 模式 | 描述 | 定价 | 实现方式 |
|------|------|------|----------|
| 大群体验 | 用户加入大群，@机器人进行基础健康问答 | 免费 | 企微群聊机器人 |
| 1v1 专属群 | 付费会员专属群，机器人 + 人工客服 | 会员专享 | 企微应用私聊 |
| 私聊对话 | 用户直接与机器人私聊 | 免费/会员 | 企微应用私聊 |

#### 3.1.4 消息类型

| 类型 | 触发场景 | 处理方式 | 企微 API |
|------|----------|----------|----------|
| 文本消息 | 用户输入问题/指令 | AI 对话 / 业务处理 | 回调 XML 解析 |
| 图片消息 | 用户发送图片 | OCR 识别 / 视觉理解 | 回调 XML 解析 |
| 文件消息 | 用户上传体检报告 | 异步解析处理 | 回调 XML 解析 |
| 事件消息 | 用户加入/退出群 | 身份识别/引导 | 回调 XML 解析 |

### 3.5 客服审核工作流

#### 3.5.1 设计理念

利用企微消息实现客服审核，无需额外开发后台界面。客服通过企微就能接收审核任务、查看问题、编辑回复、发送结果，全部在企微内完成。

#### 3.5.2 流程设计

```
用户提问 → AI 生成回答 → 分类判断
                            │
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
        L0 直接回复    L1/L2/L3 进入审核队列
                                    │
                                    ▼
                    企微消息推送给客服
                                    │
                                    ▼
                    客服手机收到消息
                                    │
                                    ▼
                    客服查看并回复（编辑内容）
                                    │
                                    ▼
                    消息推送给用户
```

#### 3.5.3 审核消息推送格式

```markdown
【待审核问题】
用户：{user_name}
时间：{timestamp}
级别：{review_level}

用户提问：
{user_message}

AI 生成回答：
{ai_response}

请回复以下格式进行审核：
- 通过：直接回复 "通过"
- 修改后通过：回复 "通过" + 你的修改内容
- 驳回：回复 "驳回" + 驳回原因
```

#### 3.5.5 审核级别与消息推送

| 级别 | 问题类型 | 推送方式 | 紧急程度 |
|------|----------|----------|----------|
| L0 | 基础健康知识、定义解释 | 不推送，直接回复 | - |
| L1 | 饮食建议、运动方案 | 推送给客服（可选抽检） | 低 |
| L2 | 科室推荐、医院选择 | 必须推送客服审核 | 中 |
| L3 | 诊断建议、用药指导 | 必须推送客服确认 | 高 |

#### 3.5.6 审核操作指令

| 指令 | 格式 | 系统行为 |
|------|------|----------|
| 直接通过 | 通过 | 推送 AI 原回答给用户 |
| 修改后通过 | 通过 {修改后的内容} | 推送修改后的内容给用户 |
| 驳回 | 驳回 {原因} | 标记为已驳回，不推送回答给用户 |
| 转人工 | 转人工 {说明} | 标记为需人工客服介入 |
